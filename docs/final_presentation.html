Project Overview

CodeMap
AI 코드베이스 온보딩 도우미

GitHub 저장소 URL 하나로 저장소 구조, 핵심 파일, 실행 흐름, 코드 근거를 분석하고 신규 개발자가 바로 이해할 수 있는 대화형 온보딩 경험을 제공하는 서비스입니다.

기획 배경

정적 분석과 일반 챗봇의 사이

README만으로는 실제 진입점, 의존성, 실행 방법을 파악하기 어렵고, 단순 챗봇은 근거 파일과 라인 추적 없이 부분 답변에 머무르는 문제가 있었습니다.

프로젝트 목적

능동적 지식 분석가

Planner, Dispatcher, Worker, Evaluator가 협력하여 코드베이스를 탐색하고, 검색 실패 시 재탐색 근거를 남기는 Agentic RAG 구조를 구현했습니다.

기대 효과

온보딩 시간과 피로도 절감

프로젝트 구조 요약, 파일 읽기 순서, 라인 단위 근거 칩, 분석 대시보드를 통해 새로운 코드베이스 파악 비용을 줄입니다.

주요 사용자 스토리
  • 사용자는 GitHub URL을 입력해 저장소 분석 작업을 생성한다.

재탐색 판단
  • LLM은 계획과 최종 답변에 집중하고, 파일 접근·보안 검증은 결정론적 코드로 분리했습니다.
  • Hybrid Search는 pgvector 의미 검색과 BM25/RRF 키워드 검색을 결합합니다.
  • Evaluator는 근거 충분성 판단 후 설정된 반복 한도 안에서 re-plan을 요청합니다.

Database Modeling

DB 모델링: 분석 작업, 코드 지식, 대화 기억

users

email, hashed_password, refresh_tokens와 연결되는 인증 기준 테이블

analysis_jobs

repo_url, owner, branch, status, stage, progress, report_json, visibility, last_accessed_at

code_nodes

job_id, path, type, depth, content, summary, embedding vector(1536), metadata

code_dependencies

source_id, target_id, relation으로 파일 간 import 그래프 저장

docs

repo_id, job_id, doc_type, content, version, is_active, report_json

chat_conversations

repo_id 기준 대화 스레드와 title, created_at, updated_at 보존

chat_messages

conversation_id, role, content, mode, references JSONB로 근거 복원

checkpoints

LangGraph AsyncPostgresSaver가 thread_id별 실행 상태 저장

teams

team_members, team_invites와 함께 팀 workspace 공유 범위 확장

검색 성능

`code_nodes.embedding`에 HNSW cosine index를 적용해 1536차원 임베딩 검색을 빠르게 수행합니다.

중복 방지

`job_id + path + chunk_index` 유니크 제약과 in-progress 부분 유니크 인덱스로 중복 분석을 제어합니다.

복원성

채팅 메시지, references, checkpoints를 분리 저장해 UI 복원과 Agent 실행 복원을 동시에 지원합니다.

Feature 2

지능형 채팅 UX 및 독립 세션 복원

채팅 UX
  • Agent 단계는 `AgentExplorationTimeline`으로 표시해 사용자가 “무엇을 탐색 중인지” 볼 수 있습니다.
  • AI 답변은 Markdown, Mermaid, 코드 블록을 렌더링하며 스트리밍 중 커서 상태를 표시합니다.
  • `references` 이벤트는 파일명, 라인 범위, 전체 경로, snippet을 가진 근거 칩으로 변환됩니다.
  • Chrome 번역으로 메시지 DOM이 변형되는 문제를 막기 위해 `notranslate`와 `translate="no"`를 적용했습니다.
독립 세션 복원
  • `repoId + threadId` 변경을 감지하면 렌더 단계에서 이전 메시지를 즉시 초기화해 세션 누수를 차단했습니다.
  • 서버에 저장된 thread가 있으면 `fetchThread`로 우선 복원하고, 없을 때 프로젝트별 localStorage를 fallback으로 사용합니다.
  • 백엔드는 `chat_conversations`, `chat_messages`, `sessionId`, LangGraph `thread_id`로 대화 맥락을 이어갑니다.
  • 최근 PR #317, #310에서 프로젝트 전환 시 이전 대화가 남는 문제를 해결했습니다.

Feature 4

Agent 방어 로직 및 파이프라인 확장

환각·무한 루프 방어
  • Evaluator는 근거가 부족할 때만 `replan_started`를 발생시키며 `AGENT_MAX_REPLANS`는 0~3 범위로 제한됩니다.
  • `attempted_signatures`를 state에 누적해 0건 반환 탐색을 반복하지 않도록 차단했습니다.
  • Dispatcher는 allowlist와 path traversal 검증을 거친 plan만 worker로 전달합니다.
  • Grep Worker에는 safe regex timeout guard를 적용해 ReDoS성 패턴으로 인한 지연을 방지했습니다.
파이프라인 확장
  • Clone → Parse → Embed → Report → Docs 단계로 저장소 분석 파이프라인을 구성했습니다.
  • AST 품질 도구, 환경 검증 도구, Hybrid Search, 문서 생성 API가 독립 도메인으로 확장되었습니다.
  • 온보딩 가이드 생성, 파일 단위 요약, 추천 읽기 순서, PDF 다운로드 UI까지 산출물 흐름이 연결됩니다.
  • 최근 발표 안정화에서는 분석 페이지 localStorage SWR 캐시를 적용해 DB/API 반복 요청 의존도를 낮췄습니다.

Retrospective

KPT 회고 및 결론

Keep
  • 프론트/백엔드 모두 도메인 중심 구조를 유지해 역할 충돌을 줄였습니다.
  • PR 리뷰와 명세 문서를 기준으로 구현을 검증해 계약 불일치를 빠르게 찾았습니다.
  • 근거 칩, Agent 타임라인, 코드 프리뷰처럼 사용자가 검증 가능한 UX를 지향했습니다.
Problem
  • Agent가 근거 부족 상황에서 같은 탐색을 반복하거나 None/set 병합 오류로 실패하는 이슈가 있었습니다.
  • 프로젝트 전환 시 이전 분석 화면과 채팅 세션이 남는 라우터·상태 동기화 문제가 있었습니다.
  • 배포 컨테이너의 `git` 누락처럼 로컬과 운영 환경 차이에서 Clone 실패가 발생했습니다.
Try
  • Evaluator 재탐색 한도, attempted signatures, safe regex 등 방어 로직을 더 체계화합니다.
  • RDB 대화 이력과 LangGraph checkpoint의 중복 적재를 일원화해 장기 기억 구조를 단순화합니다.
  • 향후 PR 자동 리뷰, 변경 영향도 분석, 팀 workspace 권한 모델을 고도화합니다.
최종 결론

CodeMap은 단순 코드 검색기가 아니라, 저장소를 분석하고 근거를 제시하며 온보딩 산출물까지 만드는 “검증 가능한 AI 코드베이스 안내자”를 목표로 구현되었습니다.
